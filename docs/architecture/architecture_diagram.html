Block legend

BLOCK 01
Attack Simulation
Atomic Red Team, Caldera, custom ransomware simulator. Source of controlled positive cases mapped to MITRE TTPs.
BLOCK 02
Victim Lab
Two virtualized hosts (Windows + Linux) with Sysmon/auditd telemetry. Canary files deployed in strategic paths.
BLOCK 03
Wazuh Manager
Single point for event collection and normalization. OpenSearch as storage backend.
BLOCK 04
Detection — 3 parallel layers
Rule-based Sigma + Unsupervised ML anomaly + Deception canaries. Each layer fails independently of the others.
BLOCK 05
SOAR Decision Engine
Score fusion logic. Combines outputs from all detection layers into severity + automated action.
BLOCK 06
LLM Triage (Layer 4)
FastAPI + mini-RAG + LLMClient (DeepSeek primary, Qwen fallback). Structured analysis enrichment.
BLOCK 07
Automated Response
Host isolation, process kill, disk snapshot, team notification. Triggered by Decision Engine.
BLOCK 08
Visualization & Metrics
OpenSearch Dashboards for SOC view, Streamlit for analyst triage UI, three metric categories: demo, forensics, evaluation.

A ransomware incident, step by step

01
Attack initiationAtomic Red Team executes T1486 (Data Encrypted for Impact) on the Windows victim. Custom Python ransomware enumerates and begins encrypting files.
02
Telemetry captureSysmon logs file events, process creations, registry changes. Wazuh agent forwards to manager. Canary file is touched within seconds.
03
Parallel detectionLayer 3 (canary) fires immediately — high confidence. Layer 1 matches T1486 Sigma rule. Layer 2 ML detects entropy spike + crypto syscall pattern.
04
Decision fusionSOAR Decision Engine receives all three signals. Logic: Layer 3 + corroboration → maximum severity → immediate isolation playbook.
05
LLM enrichmentFastAPI receives full context. RAG retrieves MITRE T1486 + NIST 800-61 ransomware playbook. DeepSeek-V3 generates structured analysis with technique, severity, recommended action.
06
Automated containmentHost isolated via firewall rules. Offending PID killed. Disk snapshot captured for forensics. Team notified via Slack with LLM analysis attached.
07
Analyst reviewStreamlit UI shows alert + LLM analysis side-by-side with citations. OpenSearch dashboard updates MITRE coverage heatmap and time-to-detect histogram.
// human-in-the-loop SOAR

Confidence-tiered automation with approval flow

Not every alert deserves automated containment, and not every alert can wait for human approval. ARGOS classifies alerts into four tiers based on detection confidence, and routes them differently — automatic for high-confidence cases, human-in-the-loop with email approval and split-brain resolution for uncertain cases. Full rationale in ADR-0003 and ADR-0006.

TIER T0 — CRITICAL
≥ 0.95 confidence
Trigger: Layer 3 alone, or Layers 1+2+3 together.
Action: auto-isolate immediately.
Email: post-facto, with "Revert if false alarm" button.
TIER T1 — HIGH
0.80 – 0.95 confidence
Trigger: Layer 1 + Layer 2 corroborate.
Action: auto-isolate + disk snapshot.
Email: post-facto with "Revert" button.
TIER T2 — MEDIUM
0.60 – 0.80 confidence
Trigger: Layer 1 alone (high-fidelity), or Layer 2 alone (very high score).
Action: awaiting approval, 3-min countdown, escalates to T1 on timeout.
Email: pre-approval with Approve / Reject buttons.
TIER T3 — LOW
0.40 – 0.60 confidence
Trigger: Layer 2 medium score, or Layer 1 experimental rule.
Action: notification only, no execute button.
Email: LLM analysis to analyst for review.
// split-brain resolution

When approvers disagree

Multiple approvers can give contradictory responses to the same email — the classic distributed consensus problem with humans as nodes. ARGOS resolves this with conservative-wins policy and a 60-second consolidation window. In containment context, conservative means isolate, because a falsely-isolated host has lower cost than a real attack uncontained.

A
First positive response opens the windowFirst click — whether "approve" or "reject" — starts a 60-second consolidation window. Subsequent responses are tracked in the audit log.
B
Same-sign confirmationsIf all subsequent responses align with the first, decision confirmed. No conflict.
C
Opposite-sign response → conflict detectedSystem logs conflict. Live banner in Approval Workflow Console: "CONFLICT DETECTED · 60s consolidation window in progress".
D
Window closes → conservative-wins appliedIf at least one approve exists, isolation executes — regardless of how many rejects were cast. Audit log records: "EXECUTED: 2 approve, 1 reject, conservative-wins policy".
E
Final email + dashboard updateAll approvers receive audit summary. Approvers who rejected can see the policy justification. Reverting requires logging into the dashboard (not via email) — adds friction to mistaken reverts.

// resilience by design

What happens when things break

A defensive system that fails silently is worse than no system at all. ARGOS is designed against failures of its own components and against adversaries actively attacking the defender. The full STRIDE + FMEA threat model (≈40 threats and failure modes) is in THREAT_MODEL.md.

R-01
LLM never on critical path
If LLM hallucinates or goes down, SOAR containment still triggers from Layers 1-3. The LLM only enriches the analyst's view — it cannot prevent isolation, only fail to explain it.
R-02
No SPOF in detection
Three layers run in parallel. Failure of any one degrades coverage; it does not blind the system. Layers 2+3 cover Layer 1's blind spots and vice versa.
R-03
Agent disconnect = signal
If an attacker kills the Wazuh agent on a victim, the heartbeat loss is itself a high-priority alert (rule 502). Suspicious silence becomes a positive indicator of compromise.
R-04
Logs ship in real time
Telemetry forwards immediately, never buffered locally. An attacker clearing local logs after action cannot prevent events from being already indexed remotely (mitigation against MITRE T1070.001).
R-05
LLM output validated
Every response: Pydantic schema check, MITRE technique whitelist (rejects hallucinated IDs), confidence threshold, no-action constraint. LLM cannot trigger isolation or kill — output is descriptive only.
R-06
Containment fails closed
If automated isolation playbook fails, the alert state remains "uncontained" and escalates to manual response. There is no scenario where a failed action silently appears successful.
R-07
Vendor portability
DeepSeek primary → Qwen fallback → continue without enrichment if both down → swap to local Llama with one config change. No vendor lock-in failure mode.
R-08
Cost-bounded operations
LLM API calls have per-incident rate limits and monthly budget caps. Pathological attack patterns or runaway bugs cannot drain the budget — system fails closed on cost before exceeding the cap.

Critical failure scenarios

!
LLM API completely downAlerts continue without enrichment. Analyst sees raw alert with full telemetry. SOAR containment unaffected. Demo: visible failover Qwen → narrate "this is where LLM analysis would appear" with screenshot if both fail.
!
LLM hallucinates non-existent MITRE techniquePydantic + ATT&CK whitelist rejects response. Alert proceeds without enrichment. Validation layer prevents bad analysis from reaching the analyst.
!
Attacker kills Wazuh agent on victimHeartbeat loss within 60s → high-priority "agent stopped" alert → host investigated as compromised. Sysmon continues local logging. Optional Filebeat secondary shipper to backup collector.
!
Wazuh manager service crashessystemd auto-restart (~30s). External heartbeat alerts the team. Agents queue events locally during outage and ship on recovery. Manager admin actions logged externally.
!
Attacker clears Windows Event Log (T1070.001)Events already shipped to manager in real-time. Log clearing event itself fires a Sigma rule. Attacker cannot retroactively delete what's already indexed remotely.
!
Race: attack faster than detectionDocumented theoretical TTD floor of 3-5 seconds. Some files will encrypt before isolation. Honest metric reported in evaluation, not hidden.
!
Live demo VM crash or network failurePre-recorded high-quality video as last-resort backup (recorded day before). Demo runs entirely on internal lab — no internet dependency for core flow. Snapshot of pre-attack state for VM rollback.
