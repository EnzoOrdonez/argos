A.Qué es ARGOS + estado por parte

ARGOS es un XDR multivector que defiende la base de datos del banco ficticio IntiBank. Combina 4 capas de detección — Sigma (L1), ML (L2), canarios (L3) y triage por LLM (L4, Capa 4) — con un orquestador SOAR + aprobación humana (HITL). Las acciones destructivas (aislar un host) pasan por tiers (T0 auto-ejecuta sub-segundo; T1–T3 esperan votos) con two-person rule y conservative-wins. Invariante R-2: el LLM nunca bloquea la contención (si falla, degrada a None y el flujo sigue).

B.Cambios y decisiones por fase (1–6) + ADRs

El prototipo funcional se construyó por fases, sin tocar argos_contracts/ (inmutable v1.1.0) ni soar/ (ya cerrado), conmutando solo los bordes (feeder y executor) por entorno.

FaseQué se construyóADR / nota
F1 — Modo en vivo demo_injector --live (deja el incidente en AWAITING_APPROVAL, la ventana de 60s arranca con el 1er voto) + live_approve.py (trigger local sin internet) + Telegram real vía ngrok. Camino A del bridge realiza ADR-0014 (events:normalized).
F2 — Capa 2 ML Scoring de la alerta normalizada por rule.mitre.id + whitelist de técnicas + override conservador. Publica al mismo stream que el bridge. Anti-alucinación / validación coherente con el contrato.
F3 — Active-response Scripts para los 2 OS del lab: bash (iptables/kill/tar/renice) en las víctimas Linux, PowerShell (netsh/Stop-Process/Copy-Item) en la Windows. Whitelist del manager (1514/1515) antes del block-all. ADR-0015 §2.3 (enmienda 2026-06-26: Win y Linux).
F4 — Triage LLM (Capa 4) Cliente OpenAI apuntado a NVIDIA NIM (deepseek-v4-pro primario, kimi-k2.6 fallback), POST /triage, fail-soft a None (R-2), sanitizer T-030 antes de la nube, validación MITRE whitelist (R-6). ADR-0001 v3 (reconcilia soberanía — ver abajo).
F5 — docker-compose Servicios core en la VM Linux core (redis, postgres argos_audit, soar, console, llm-triage) + perfiles real (bridge) y fallback (streamlit). Default ARGOS_EXECUTOR=simulated. ADR-0015 (enmienda 2026-06-26b: Perfil A).
F6 — Consola web Consola read-only (:8080) que refleja el HITL en vivo (incidente, matriz de aprobadores, reloj, banner de decisión/split-brain, panel LLM). La Streamlit (:8501) queda de fallback. Lee incident:{id} de Redis; no toca el SOAR.

Reconciliaciones clave (decisiones que parecían contradecirse y se documentaron honesto)

  • ADR-0001 v3 — soberanía del LLM. v2 descartó DeepSeek por jurisdicción PRC. v3 usa deepseek-v4-pro/kimi-k2.6 vía NVIDIA NIM: el dato va a NVIDIA (US), no a servidores PRC; procedencia del modelo ≠ jurisdicción del dato. Defensa: data sintética + sanitizer T-030 + Ollama zero-egress (diferido).
  • ADR-0011 — contrato > manual. Emitir payload (NO data); outcomes reales del contrato (EXECUTE_ISOLATION/NO_ACTION, no "execute"/"block" del manual viejo).
  • ADR-0010 §4.5 — Postgres 17.5-bookworm (no el 15 del manual; no-negociable).
  • ADR-0015 — topología. 3 VMs; AR Win+Linux; Perfil A manager-only (sin OpenSearch); Wazuh instalado en la VM core (no servicio docker); argos_audit en el compose core, separado del app_prod de la víctima; ml es librería, no servicio.

C.Prerrequisitos + qué levanta cada máquina

Software

  • Python 3.11 (canónico; pydantic-core no compila en 3.14).
  • Docker + docker compose v2 (para el core / Redis). Sin Docker: Memurai o Redis en WSL.
  • ngrok (solo para aprobar por Telegram en vivo, en la máquina del Approval API).
  • Key NVIDIA en .env como OPENAI_API_KEY (para la Capa 4 real). Sin ella, el triage degrada a None — el demo igual corre.
  • Bot de Telegram (token + chat id) — opcional, para notificaciones reales.

D.Comandos exactos (en orden)

Camino simulado — el del video, garantizado (una sola PC, sin lab)

Opción A — docker-compose (todo el core junto):

# raíz del repo. .env con POSTGRES_PASSWORD (+ OPENAI_API_KEY si querés el LLM real).
docker compose up -d                 # redis, postgres, soar, console, llm-triage
docker compose ps                    # esperar (healthy)
python scripts\demo_injector.py uc04 --redis-url redis://localhost:6379/0
# consola web -> http://localhost:8080

Opción B — manual (3 terminales, sin Docker para los servicios):

# T1 — Redis
docker run --rm -p 6379:6379 redis:7
# T2 — consola (Streamlit fallback)
$env:REDIS_URL = "redis://localhost:6379/0" ; streamlit run ui\streamlit_app\app.py   # :8501
# T3 — inyectar (NO --in-process: usa el Redis real)
.\.venv\Scripts\python.exe scripts\demo_injector.py uc01 --redis-url redis://localhost:6379/0

Camino live — teatro en vivo (reloj de 60s + aprobar en el momento)

Trigger local (sin internet, la red de seguridad):

# 1. inyectar dejando el incidente EN ESPERA
.\.venv\Scripts\python.exe scripts\demo_injector.py uc04 --live --redis-url redis://localhost:6379/0
#    -> imprime el incident_id (INC-AAAA-MM-DD-NNN)
# 2. votar (two-person: 2 approves DISTINTOS ejecutan; 1 reject cancela)
.\.venv\Scripts\python.exe scripts\live_approve.py --latest --decision approve --email telegram:soc-lead
.\.venv\Scripts\python.exe scripts\live_approve.py --latest --decision approve --email telegram:dba
#    rechazar:  --decision reject --email telegram:compliance

Telegram + ngrok (una vez, en la máquina del Approval API):

$env:REDIS_URL = "redis://localhost:6379/0"
.\.venv\Scripts\python.exe -m uvicorn soar.approval_api.main:app --port 8003
ngrok http 8003          # copiar la URL https://XXXX.ngrok-free.app
# registrar el webhook (navegador):
# https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://XXXX.ngrok-free.app/telegram/callback
ngrok cambia de URL en cada reinicio → repetir el setWebhook con la URL nueva. Para el camino Telegram, correr el Approval API sin ARGOS_JWT_SECRET (Fase 1 = modo legacy sin token firmado) y agregar --no-wait al live_approve (el API es dueño del reloj).

Swap simulado↔real de la contención: $env:ARGOS_EXECUTOR = "wazuh" (degrada a simulated si faltan las WAZUH_API_*). Con compose: --profile real suma el bridge.

F.Troubleshooting

SíntomaCausa / fix
UnicodeEncodeError: charmapSolo en los simuladores de P3. $env:PYTHONUTF8 = "1" antes de correr (los 2 tests uc06/uc08 CLI default mode que fallan son por esto, no por lógica).
Consola vacíaEl inyector tiene que correr con --redis-url (NO --in-process, que usa un fakeredis aislado) apuntando al mismo REDIS_URL que la consola.
Puerto 6379 ocupadoHay otro Redis/contenedor (p. ej. un docker run --rm redis viejo). Pararlo o usar el del compose.
BUSYGROUP ... already existsInofensivo. Reset: redis-cli XGROUP DESTROY events:normalized soar-router + XGROUP CREATE events:normalized soar-router 0 MKSTREAM.
ngrok no recibe el clickLa URL cambió al reiniciar → repetir el setWebhook. El webhook es excluyente con getUpdates.
Triage LLM no respondeLa key real está en .env bajo DeepSeek_V4_PRO_API_KEY/Kimi2_6_API_KEY; copiala a OPENAI_API_KEY. Igual es fail-soft (None) — no bloquea la contención.
ModuleNotFoundError: openpyxlFalta el extra; pip install -e ".[soar,llm,dev,ui]".
Docker no instaladoMemurai / Redis en WSL, o --in-process (pero la consola no lo verá).

G.Quién hace qué en el prototipo real

Una frase: un bot, un ngrok, un Approval API; los demás solo aprueban o levantan su parte de infra.

  • P4 (Diego) — el lab. Hostea las 3 VMs, instala el Wazuh manager en la core, provisiona app_prod + argos_audit (postgres 17.5), instala los agentes Wazuh + scripts AR en las víctimas, y ejecuta el ataque.
  • P1 (Enzo) — el SOAR. Crea el bot de Telegram una vez; corre el Approval API + ngrok en una sola máquina.
  • P2 (Sebastián) — ML/bridge. El bridge tailea el alerts.json del manager; el scorer ML publica al stream.
  • P3 (Ángeles) — detección/AR. Reglas Sigma + canarios desplegados en el manager; scripts AR en las víctimas.
  • Los que solo aprueban: estar en el chat de Telegram y tocar el botón. No instalan ni configuran nada.

H.Trampas de integración (no romper la coherencia)

  • Emitir payload, NO data — el campo del mensaje en events:normalized es payload (NormalizedAlert.model_dump_json()). Es el error de integración #1.
  • Seguir los ADRs, no los manuales viejos — si un manual/PDF dice algo distinto (outcomes "execute"/"block", postgres 15, OpenSearch en Perfil A), gana el ADR / este documento.
  • Postgres 17.5-bookworm (ADR-0010 §4.5), no el 15 del manual.
  • Perfil A = manager-only — sin OpenSearch/Dashboards (eso es Perfil B). Wazuh manager instalado en la VM core, no como servicio del compose.
  • argos_audit ≠ víctima — el log de ARGOS corre en la core, separado del app_prod que defiende (no en la caja comprometible).
  • No tocar argos_contracts/ ni soar/ — son inmutables/cerrados; conmutar solo los bordes (ARGOS_EXECUTOR, feeder demo_injector ↔ bridge).
  • R-2 — el LLM nunca bloquea la contención (fail-soft a None).

F7 (Kubernetes, Perfil B) queda diferido como artefacto futuro: stack distribuido con wazuh-indexer + OpenSearch Dashboards (los 3 dashboards SOC) + k8s, pendiente de ADR-0016. No es parte del prototipo F1–F6.
